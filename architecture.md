Ты — опытный архитектор программного обеспечения. Твоя задача — спроектировать **настольное приложение с графическим интерфейсом**, которое озвучивает книги (формат FB2) с помощью синтеза речи Microsoft Edge двумя голосами: один читает основной текст, второй произносит сгенерированные AI комментарии. Приложение должно легко собираться в **единый исполняемый файл** (один .exe для Windows, .app для macOS, один бинарник для Linux) для простого распространения.

**Основная концепция:**
Пользователь загружает fb2-файл, парсит его, извлекает текст по главам. После каждых N предложений (настраивается) вставляется комментарий от выбранной роли комментатора. Комментарии генерирует языковая модель через API (DeepSeek — рекомендуемый, также ChatGPT, Grok, Qwen). Итоговая аудиокнига сохраняется локально: главы объединяются в один файл.

**Функциональные требования:**

1. **Парсинг FB2**  
   - Извлечение текста и структуры глав (разделы), базовых метаданных (название, автор).  
   - Очистка от нетекстовых элементов (сноски, картинки, теги форматирования).  
   - Сохранение разбивки на предложения (с учётом особенностей русского, английского, японского, китайского языков — язык книг настраивается).

2. **Управление комментариями**  
   - Частота комментирования: после каждых N предложений (N задаётся пользователем, едино для всей книги или с разбивкой по главам).  
   - Выбор AI-провайдера: DeepSeek (рекомендуемый), ChatGPT, Grok, Qwen.  
   - Системный промпт роли комментатора: либо выбор из заготовок («Строгий критик», «Восторженный фанат», «Учёный эксперт»), либо ввод своего текста.  
   - Генерация комментария через API: отправка контекста (предыдущие несколько предложений) и получение короткого ёмкого комментария (не более 2-3 предложений).

3. **Синтез речи (Microsoft Edge TTS)**  
   - Бесплатный движок: `edge-tts` или аналог.  
   - Два раздельных голоса: для основного текста и для комментариев (выбираются из доступных в системе).  
   - Тонкая настройка: паузы между основным текстом и комментарием, темп речи.  
   - Склейка аудио: сначала все фрагменты одной главы → глава в один аудиофайл → затем все главы в полную книгу.

4. **Обработка ошибок и надёжность**  
   - При отказе API (сеть, таймаут, превышение лимита) — до 3 повторных попыток с экспоненциальной задержкой.  
   - Если комментарий не получен — вставка резервного текста («Комментарий временно недоступен») или пропуск без остановки всего процесса.  
   - Сохранение прогресса (контрольная точка после каждой главы) с возможностью продолжить после сбоя/закрытия.  
   - Итоговые временные аудиофайлы удаляются по завершении, кроме финального.

5. **Графический интерфейс (GUI) и сценарий использования**  
   Приложение должно быть **нативным десктопным окном**, а не веб-интерфейсом. Пошаговый мастер настройки:
   - *Шаг 1.* Первый запуск — выбрать язык интерфейса и язык книг (русский / английский / японский / китайский).  
   - *Шаг 2.* Выбрать AI-провайдера и ввести API-ключ (поле скрытое, ключ не отображается открыто).  
   - *Шаг 3.* Показать изображение `logo.png` (загружается из ресурсов приложения) и кнопку «Продолжить».  
   - *Шаг 4.* Загрузить FB2-файл (диалог выбора файла). Отобразить метаданные: название, автор, количество глав.  
   - *Шаг 5.* Настройка объёма озвучки: одна глава, диапазон глав (N–M), вся книга.  
   - *Шаг 6.* Задать частоту комментирования (каждые N предложений), выбрать или написать системный промпт роли.  
   - *Шаг 7.* Кнопка запуска «Создать аудиокнигу “Название”».  
   - В процессе: окно прогресса (полоса, текущая глава, процент, оценка времени). Возможность поставить на паузу / отменить.  
   - По завершении: сообщение с путём к готовому аудиофайлу.  

   Интерфейс должен оставаться отзывчивым во время длительных операций (генерация комментариев, синтез речи). Все тяжёлые задачи выполняются в фоновых потоках.

6. **Безопасность и хранение конфигураций**  
   - API-ключ хранить **безопасно**: использовать системный keyring (credential store) или зашифрованный локальный файл с привязкой к машине. Никакого открытого текста в конфигах.  
   - Настройки (язык, выбранные голоса, пути сохранения) сохранять в локальном конфигурационном файле (JSON/YAML) в папке пользователя.

7. **Целевые платформы и упаковка в один исполняемый файл**
   - Приложение должно работать и собираться в единый исполняемый файл как минимум на **Windows** и **Linux**..
   - Выбранный стек технологий должен позволять кросс-платформенную сборку из одной кодовой базы без существенных ветвлений кода.
   - Инструкция по сборке должна включать:
     * команды для создания одного .exe для Windows (например, через PyInstaller, Nuitka или аналог);
     * команды для создания одного бинарника/AppImage для Linux;
     * (опционально) .app для macOS.
   - Требуется минимизировать размер итогового файла и количество внешних зависимостей, требуемых в runtime (например, не требовать отдельно установленного Python или Edge).
   - Приоритет: нативный или близкий к нативному GUI, который не тащит браузерный движок (избегать Electron, если это приведёт к раздуванию размера).

**Технические задачи, которые нужно решить в архитектуре:**
- **Асинхронность и многопоточность:** GUI не блокируется во время сетевых запросов, синтеза речи и склейки аудио.
- **Потоковая обработка:** книга может быть большой, избегать загрузки всего текста в память; работать с чанками глав.
- **Очередь запросов к API:** управление частотой, контроль параллельных запросов (с учётом ограничений провайдера).
- **TTS интеграция:** управление голосами через edge-tts, программная генерация тишины для пауз, корректное объединение аудиосегментов без потери синхронизации.
- **Кроссплатформенность:** приложение должно работать на Windows, macOS, Linux. GUI-фреймворк должен обеспечивать нативный вид или быть кроссплатформенным с единой кодовой базой.
- **Логирование и отладка:** подробные логи ошибок, сохранение лога рядом с выходным файлом.

5. Детали реализации надёжности (retry-логика, чекпоинты, восстановление).
6. Схема безопасного хранения ключа (keyring / шифрование).
